## 3. 대시보드 데이터 파이프라인 핵심 아키텍처 (매우 중요)
만약 컨텍스트가 끊기거나 다른 기기에서 작업할 경우, 다음 로직을 **절대 잊지 말고** 구현해야 함.
1. **D-1 증분 적재 (API)**
   - API 필터의 결함(누락)을 피하기 위해, 매일 어제(D-1) 날짜의 **"전국"** 데이터를 모두 호출하여 로컬 DB(SQLite 등)에 누적(Upsert)한다.
2. **[변경됨] 수요기관 판별 방식 전면 개편: 마스터 DB 기반 절대 기준 (가장 확실)**
   - **수요기관 마스터**: `부산광역시 조달 수요기관 마스터파일_최종본.csv` (또는 `busan_agencies_master.db`)를 기준으로 삼습니다.
   - **지역업체 마스터**: `busan_companies_master.db` (1984~현재까지의 CSV 시드 3.3만건과 1990~현재 조달업체 API 호출로 구축한 부산 관내 DB). 수동 엑셀 파일(`조달업체 등록 내역.csv`, `부산_지역업체_마스터.csv`)은 전면 사용 금지.
   - **[1단계 (수요기관 판별 - 마스터 DB 필수)]**: 기존처럼 API 주소 텍스트에 '부산'이 있다고 부산 발주처로 1차 확정하는 방식을 **완전 폐기**합니다. 대신 통과되는 모든 기관의 `수요기관코드`, `사업자등록번호(bizno)`, `수요기관명`을 가지고 `수요기관 마스터 DB`와 직접 조인(조회)하여 마스터 DB에 존재하는 경우에만 '부산 관내 발주처'로 1차 스캔 판별합니다.
   - **[2단계 (조달업체 판별 - 마스터 대조)]**: 1단계에서 구별된 부산 관내 기관의 계약 건에 한해서 원수주 업체의 사업자번호를 우리 `지역업체 마스터 DB`와 조인하여 최종 관내 수주 여부를 확정합니다. 
   - **[3단계 (수요기관 마스터 자가 학습)]**: API를 통해 들어온 신생 부산 기관(마스터에 없는 경우)은 신규 갱신 로직을 통해 마스터 DB에 추가하여 지속 학습시킵니다.
## 4. 공동/분담/주계약자 지분율 추출 꼼수(Cheat-key)의 완벽 증명
- 데이터허브 **엑셀 수동 다운로드 파일은 "공동, 분담, 주계약자 방식" 등에서 각 참여업체별로 행(Row)이 중복 표출되며 총 계약금액 전체가 뻥튀기(과다 합산)되는 치명적인 오류**가 있음이 증명됨.
- 따라서 향후 **수동 엑셀 파일은 전면 사용을 금지**함.
- **[해결책]** API 응답 원본의 `corpList` 필드는 어떤 분할 계약 유형이든 "계약 1건당 1행"을 보장하며, 내부 배열 텍스트를 파싱(캐럿 `^` Split)하면 각 업체의 **환산 지분율(%)** (Index 6)과 **사업자등록번호** (Index -1)를 완벽하게 추출가능함.
- **(계산식)**: `총 계약금액 x (해당 업체의 추출된 지분율 / 100)`으로 계산하여 1원의 오차 없이 부산 소재 파이(Pie)만 중복 없이 발라낼 것.

## 5. 대규모 과거 데이터 초기 적재 (Historical Initial Load) 및 초고속 병렬 수집
- 파이프라인 도입일(예: 3월 20일) 이전에 체결된 1.1 ~ 3.19까지의 3~4달치 과거 데이터 누적분 역시 수동 엑셀로 메우면 지분율 로직이 깨짐.
- **[해결책]** 최초 구축 시 파이썬 `asyncio/aiohttp` 또는 `ThreadPoolExecutor(max_workers=10)`를 적용하여 날짜 파라미터를 Loop로 넣고, 1월~도입일까지의 "과거 데이터 전체"를 초고속 병렬 다운로드(1개월 수집 약 1.5분 소요 검증 완료)하여 형태를 통일시킴.
- 초기 적재(Historical Load)를 한 번 완료한 이후부터는 원래 기획대로 매일 새벽 크론탭으로 D-1(어제 하루치) 전국 데이터를 SQLite DB에 누적 추가(Append)함.
