# 대화맥락 - 2026년 05월 03일

## 1. 개요
- **목표**: Phase 6-G: 종합쇼핑몰 등록상품과 MAS 계약유형 분리 관리 
- **결과**: MAS 상품에 종속되었던 조달청 종합쇼핑몰 데이터 구조를 개선하고, `shopping_mall_product` 신규 스키마를 신설하여 다양한 계약유형(MAS, 제3자단가, 일반단가, 우수조달)을 체계적으로 분류 및 분리 적재하도록 구조를 고도화함.

## 2. 주요 구현 내용
1. **DB 스키마 및 뷰 설계 개편 (`migrate_chatbot_db.py`)**
   - `shopping_mall_product` 신규 테이블 도입.
   - `shopping_mall_contract_type` 필드로 계약 형태 세분화.
   - `chatbot_company_candidate_view` 뷰에서 `shopping_mall_flags_raw` 및 `shopping_mall_product_summary_raw`를 동적으로 병합 제공하도록 스키마 확장.

2. **ETL 파이프라인 정밀도 향상 (`bootstrap_from_excel.py`, `import_mas_product_api.py`)**
   - 엑셀의 "MAS여부", "우수제품여부", "계약구분" 필드를 파싱하여 `shopping_mall_contract_type` 우선순위를 산출.
   - HMAC 해싱으로 인한 매칭 실패를 제거하고 `canonical_business_no` 기반으로 직접 매칭하여 오차율 0% 달성.
   - 엑셀 내 2,549건의 종합쇼핑몰 제품군을 누락 없이 정확한 타입으로 분리 적재 성공.
   - API 동기화 시 `serviceKey` 로깅 차단 및 계약번호 해시 변환(`contract_no_hash`)으로 PII 보안성 확보.

3. **챗봇 API 법적 가이드레일 통제 (`api_server.py`)**
   - 챗봇 응답 모델에 `shopping_mall_flags`, `shopping_mall_product_summary`를 공식 허용(`ALLOWED_CANDIDATE_FIELDS` 추가).
   - `candidate_types` 내에 `shopping_mall_supplier` 항목 자동 편입 로직 구현.
   - "바로 구매 가능", "계약 가능", "수의계약 가능" 등의 법적으로 오해를 부를 수 있는 결론 지시어 반환을 원천적으로 차단하고, 객관적 팩트 기반 필드만 전달하도록 로직 단속.

## 3. 미결 사항 및 향후 과제
- `production_deployment=HOLD` 정책에 따라 본 구조 변경 건은 현재 운영(Production) 서버에 배포되지 않음. 추후 일괄 릴리즈 시 DB 마이그레이션 적용 및 `api_cache.json` 초기화 필요.
- 기존 `mas_product` 의존성(하위 호환성)은 당분간 유지되나, 추후 시스템 고도화 시 완전 폐기 및 `shopping_mall_product` 단일화 추진 권장.
